一、題目說明與本題重點

本題要求說明 testbench-DUT 系統架構，也就是測試平台如何透過固定 I/O port 連接學生設計的 drawBbox 模組。整體架構可分成上方的 testbench memory system 與下方的 DUT。Testbench 負責提供原始影像、接收輸出影像並與 golden answer 比對；DUT 則負責讀取 ImgROM、執行影像處理 pipeline，最後將 overlay 結果寫入 AnsSRAM。

核心觀念：本 Lab 不是只寫影像演算法，而是要讓演算法符合硬體記憶體介面、時脈同步、啟動與完成訊號規範。因此，drawBbox 的 port 名稱、方向、bit width 與讀寫時序都不能任意更改。

三、各區塊功能與資料流說明

1. ImgROM

ImgROM 儲存原始 RGB 影像，屬於唯讀記憶體。DUT 透過 Img_A 指定位址，並以 Img_CEN=0 啟動讀取，資料由 Img_Q[31:0] 回傳。

用途：提供原始 pixel 給灰階、濾波、二值化與 overlay 參考。

2. UrSRAM

UrSRAM 是使用者可控制的中間暫存 SRAM，可用來保存 mask、label map、暫時影像或 CCL 過程資料。此區塊需要同時支援 read/write。

用途：降低內部 register array 壓力，協助多階段 pipeline 存取資料。

3. AnsSRAM

AnsSRAM 儲存最後輸出影像。當 bounding box 位置成立時，DUT 寫入綠色框線；其他位置則保留原圖 RGB，形成最終 overlay 結果。

用途：testbench 讀取後與 bbox_golden*.txt 做 pixel-by-pixel 比對。

4. drawBbox DUT

DUT 是本作業主要設計內容，負責控制 memory timing、掃描 pixel、執行 image processing pipeline、產生 bounding box 並寫回結果。

用途：完成從輸入影像到輸出標框影像的完整硬體流程。

六、Row-major Address Mapping

本 Lab 的輸入與輸出影像大小皆為 256×256，每一個 pixel 對應一個 16-bit address。掃描順序採 row-major，也就是先掃完第 0 列的 256 個 pixel，再進入第 1 列。

Row-major mapping 示意：16-bit address 剛好覆蓋 65536 個 pixels。

九、報告可直接使用的解答文字

Fig.2 顯示本 Lab 的 testbed-DUT 架構。上方 testbench 包含 ImgROM、UrSRAM 與 AnsSRAM 三個記憶體區塊；下方則是學生實作的 drawBbox 模組。ImgROM 負責提供 256×256 的 32-bit RGB 原始影像，DUT 透過 Img_CEN 與 Img_A 送出讀取控制與位址，並由 Img_Q 接收 pixel data。UrSRAM 可作為中間資料儲存區，保存灰階、mask、label 或 CCL 過程資料。AnsSRAM 則用來儲存最後 overlay 後的影像結果，也就是保留原圖並在 bounding box 邊界寫入綠色 pixel。

在控制規則上，ImgROM、UrSRAM 與 AnsSRAM 的 CEN 皆為 active-low，因此要存取記憶體時必須設為 0。對 SRAM 而言，WEN=0 表示寫入，WEN=1 表示讀取。整體設計必須遵守同步式硬體設計原則，利用 clk 驅動 FSM 與 address counter，在 enable 啟動後開始處理影像，並在所有輸出 pixel 寫入完成後將 done 拉高，通知 testbench 進行 golden comparison。由於 testbench 會依固定 port 連接 DUT，因此 drawBbox 的 I/O port 名稱、方向與位元寬度皆不可修改。

滿分報告補強重點：說明 I/O port 時不要只列表，還要交代每個 port 在 pipeline 中何時被使用、資料方向為何、active-low/active-high 規則，以及最後如何觸發 done 讓 testbench 比對 AnsSRAM。
