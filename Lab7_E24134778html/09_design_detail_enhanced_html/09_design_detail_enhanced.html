一、題目說明

本題要求詳細描述自己的設計，可以使用內部架構圖、block diagram 或文字說明。 對 Lab7 而言，drawBbox 不只是單一模組，而是一個整合式影像處理硬體系統。 它需要控制 ImgROM 讀取原圖、儲存中間結果、執行灰階轉換與 Gaussian filter、計算 Otsu threshold、 產生 binary mask、執行 8-connected CCL、計算 bounding box，最後將綠色邊界框 overlay 回原始 RGB 圖並寫入 AnsSRAM。

本頁重點在於說明 drawBbox 內部如何以 FSM 搭配 internal memory / register arrays 將完整影像處理 pipeline 串接起來。
Fig.9-1 美化版 drawBbox internal architecture

二、整體設計概念

本設計採用 FSM-controlled multi-stage architecture。也就是說，所有影像處理階段都由 controller FSM 依序啟動， 每一階段只負責一個主要功能。此方式可避免在單一 clock cycle 中串接過長的 combinational logic，也能讓各階段中間資料被保存在對應 memory 中，方便 waveform 觀察與 debug。

整體資料路徑可整理為： 

ImgROM
  → rgb_mem
  → gray_mem
  → gauss_mem + histogram
  → Otsu threshold
  → mask_mem
  → label_mem + parent[]
  → bbox arrays
  → box_map
  → Overlay selector
  → AnsSRAM
Fig.9-2 internal memory / register array dataflow

四、硬體演算法 Pipeline

此設計將投影片要求的軟體式影像流程轉成硬體可控制的 pipeline。每個 stage 並非一定是獨立 Verilog module； 在本設計中，許多功能是整合於 drawBbox.v 內，透過不同 FSM state 與暫存陣列完成。

Fig.9-3 Algorithm pipeline implemented inside drawBbox
Pipeline 階段硬體處理方式輸出資料
Read Original Image FSM 控制 Img_CEN 與 Img_A，由 Img_Q 讀取 32-bit RGB。 rgb_mem
RGB to Grayscale 依照灰階公式將 RGB 加權轉成單一灰階值。 gray_mem
Gaussian Filter 對每個 pixel 取 3×3 鄰域，使用 kernel 1-2-1 / 2-4-2 / 1-2-1 做加權平均。 gauss_mem
Histogram / Otsu 統計 0~255 灰階分佈，掃描 threshold 找最大類間變異數。 threshold
Mask + Polarity 根據 threshold 產生 foreground/background，必要時執行 auto-polarity correction。 mask_mem
8-connected CCL 檢查 NW、N、NE、W 鄰居，分配 label 並使用 union-find 合併等價 label。 label_mem, parent[]
Bounding Box 掃描 label map，更新每個 component 的 Xmin/Xmax/Ymin/Ymax。 bbox arrays
Overlay / Draw 根據 box_map 選擇輸出原始 RGB 或綠色 32'h0000_FF00。 AnsSRAM

五、Memory Interface 架構

本設計與 testbench 的溝通主要透過三個 memory interface：ImgROM、UrSRAM、AnsSRAM。 ImgROM 是只讀 memory，負責提供原始 RGB pixel；UrSRAM 可用於中間資料；AnsSRAM 是最後輸出 memory。

Fig.9-4 Memory interface design and row-major mapping
影像位址採用 row-major mapping：addr = row × 256 + col。由於影像大小為 256×256，因此 address 範圍為 0～65535，需要 16-bit address。

六、CCL 與 Bounding Box 架構說明

CCL 是本設計中最重要且最容易出錯的部分。Lab7 要求使用 8-connectivity，而不是前一個 Lab 常見的 4-connectivity。 在 raster scan 的 pass1 中，只能檢查已掃描過的鄰居，因此本設計檢查 NW、N、NE、W。 若四個鄰居都沒有 label，則分配新 label；若只有一個 label，則沿用；若有多個不同 label，則建立 equivalence relation，並透過 union-find 的 parent[] 進行合併。

完成 CCL 後，系統掃描 label_mem，對每個 label 更新 Xmin、Xmax、Ymin、Ymax。 若物件接觸影像邊界，依 Lab7 規則不畫 bounding box；若物件有效，則依座標產生 box_map。 最後 overlay 階段讀取 box_map，在邊界位置輸出綠色，其餘位置輸出原始 RGB。

七、Architecture-Oriented Debug 方法

此設計將中間結果分別存入不同 memory，使 debug 可以逐階段進行。 如果直接觀察最後 BMP，很難知道錯誤來自 threshold、mask、CCL 還是 bbox。 因此本設計可依照 internal architecture 對每一階段進行檢查。

八、設計優點

1

階段清楚

每個影像處理步驟都有對應資料結構，流程從 ImgROM 到 AnsSRAM 一目了然。

2

容易除錯

gray、gauss、mask、label、bbox 皆可個別檢查，快速定位錯誤來源。

3

降低 critical path

透過 FSM 分段處理，避免將 Gaussian、Otsu、CCL 與 Draw 串在同一 cycle。

4

結果可重現

所有輸出由固定 memory 與 deterministic FSM 產生，利於 P1~P4 pixel-perfect verification。

九、報告可直接貼上的正式答案

本設計以 drawBbox 作為 top module，內部採用 FSM 控制整體影像處理流程，並搭配多組 internal memory / register arrays 儲存各階段中間結果。 系統首先透過 ImgROM interface 讀取 256×256 RGB 影像，並將原始 pixel 暫存在 rgb_mem，供最後 overlay 階段使用。 接著將 RGB 轉為 grayscale，存入 gray_mem；再執行 3×3 Gaussian filter，結果存入 gauss_mem，同時建立 histogram 供 Otsu threshold 使用。

完成 threshold 計算後，系統依據 threshold 產生 binary mask，並透過 auto-polarity correction 確保 foreground 表示目標物件。 接著 CCL 階段使用 8-connected labeling，檢查 NW、N、NE、W 四個已掃描鄰居，並利用 union-find 的 parent[] 記錄與合併 label equivalence。 CCL 完成後，系統掃描 label_mem，對每個 connected component 計算 Xmin、Xmax、Ymin 與 Ymax， 形成 bounding box。

最後，系統根據 bounding box 座標產生 box_map，若目前 pixel 位於 box 邊界則輸出綠色 32'h0000_FF00， 否則輸出原始 RGB pixel，並將最終影像寫入 AnsSRAM。此架構使每個演算法階段都具有明確輸入與輸出，方便 waveform 觀察與 pixel-level debug， 也能避免過長 combinational path，有利於後續 synthesis timing closure。
