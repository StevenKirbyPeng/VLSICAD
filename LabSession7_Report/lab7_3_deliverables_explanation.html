一、Deliverables 整體意思

這張附件主要是在說明 Lab7 作業最後上傳時必須包含哪些檔案、哪些檔案不能放進報告或壓縮檔，以及壓縮檔內部資料夾階層必須如何排列。Lab7 的重點不是只繳交 drawBbox.v，而是要讓助教能在 SoC Lab 環境中直接編譯、模擬、合成與檢查 SuperLint。因此所有必要的 Verilog、測試平台、合成腳本、限制檔、結果圖片與 PDF 報告都要完整放入指定資料夾中。

核心原則：助教解壓縮後，應該可以直接依照 Makefile 或指定指令執行 RTL simulation、synthesis、post-synthesis simulation 與 SuperLint 檢查。如果缺少任何自訂 submodule，就可能導致無法編譯，嚴重時會被視為 0 分。

二、六項提交規定逐條說明

原規定 中文詳細說明 提交重點
1) All Verilog codes including testbenches, .bmp and .txt should be uploaded. 所有 Verilog 設計檔與測試檔都要上傳，包含主程式、子模組、testbench、模擬產生或比對需要的 BMP 圖片與文字檔。這代表只放主模組是不夠的，凡是模擬會用到的檔案都應一起放入壓縮檔。 必放 Verilog、testbench、BMP、TXT。
2) Please DO NOT include source code in the paper report. 紙本或 PDF 報告中不要整份貼上 Verilog 原始碼。報告應著重在架構、流程、FSM、模組功能、實驗結果與分析。如果真的要展示程式，只能放少量關鍵片段截圖或簡短說明，不應整份程式碼塞進報告。 報告不可整份貼 code。
3) Please DO NOT upload waveforms (.fsdb or .vcd). 不要上傳波形檔，例如 .fsdb 或 .vcd。這些檔案通常很大，只是用來除錯，不屬於最後提交內容。若放入可能造成壓縮檔過大或不符合規定。 不要放 FSDB / VCD。
4) If you upload a dead body which we can’t even compile, you will get NO credit! 「dead body」是指助教拿到後完全不能編譯的作業。如果檔案缺漏、module 名稱不一致、port 對不上、語法錯誤、少放 submodule，導致無法 compile，可能直接沒有分數。 提交前一定要重新解壓縮測試 compile。
5) All Verilog file should get at least 90% SuperLint Coverage. 所有 Verilog 檔案都需要通過 SuperLint 檢查，且 coverage 至少要達到 90%。這代表除了功能正確，也要注意硬體設計風格，例如避免 latch、未使用訊號、寬度不匹配、不可合成語法等問題。 SuperLint ≥ 90%。
6) All homework requirements should be uploaded in this file hierarchy... 所有作業檔案必須依照指定階層放置。如果使用額外子模組，必須一起放入 tar 檔案；如果沒有放，助教編譯時找不到 module，就會被視為未完成或 0 分。 檔名、資料夾、階層都要照規定。

三、標準提交檔案階層

附件中的壓縮檔名稱為 Lab7_StudentID.tar。其中 StudentID 要改成自己的學號，例如 Lab7_E24134778.tar。壓縮檔內部建議放一個同名資料夾，資料夾內包含下列檔案。

Lab7_StudentID.tar
└── Lab7_StudentID/
    ├── drawBbox.v
    ├── tb.sv
    ├── CYCLE_MAX.sv
    ├── drawBbox_syn.v
    ├── drawBbox_syn.sdf
    ├── DC.sdc
    ├── .synopsys_dc.setup
    ├── superlint.tcl
    ├── synthesize.tcl
    ├── Makefile
    ├── drawBbox_result_p4.bmp
    ├── StudentID.pdf
    └── additional_submodules.v
檔案名稱 用途說明
drawBbox.v Lab7 的主 RTL 設計檔，負責影像讀取、灰階、濾波、threshold、mask、CCL、bounding box 與 overlay 控制。
tb.sv 測試平台，用來餵入測試圖、驅動 clock/reset/enable、檢查輸出結果是否與 golden answer 一致。
CYCLE_MAX.sv 設定最大模擬 cycle 或 clock 相關參數，避免設計卡住造成無限模擬。
drawBbox_syn.v 合成後的 gate-level netlist，用於 post-synthesis simulation。
drawBbox_syn.sdf 合成後的延遲標註檔，用來在 gate-level simulation 中加入實際延遲資訊。
DC.sdc Design Compiler 使用的 timing constraint 檔，通常包含 clock period 與相關限制。
.synopsys_dc.setup Synopsys Design Compiler 環境設定檔，協助工具找到 library 與相關設定。
superlint.tcl 執行 SuperLint 檢查用的 Tcl 腳本。
synthesize.tcl 合成流程腳本，通常包含讀入 RTL、設定 current design、link、compile、輸出 report 等指令。
Makefile 提供統一指令執行 RTL simulation、synthesis、post-simulation、SuperLint 與 clean。
drawBbox_result_p4.bmp Pattern 4 的輸出結果圖片，用來展示 bounding box 是否正確畫在輸出影像上。若老師要求 P1–P4 結果，也可一併附上。
StudentID.pdf 正式報告 PDF，檔名要改為自己的學號，例如 E24134778.pdf。
additional_submodules.v 如果你的設計有使用額外子模組，例如 rgb2gray_unit.v、gaussian_unit.v、otsu_unit.v、ccl_unit.v 等，必須全部放入。

四、哪些檔案一定要放，哪些不要放

1、一定要放

  • 主 RTL：drawBbox.v
  • 所有自訂 Verilog submodules：例如灰階、Gaussian、Otsu、mask、CCL、bbox、overlay 等模組。
  • 測試平台與設定檔：tb.sv、CYCLE_MAX.sv。
  • 合成與驗證腳本：Makefile、synthesize.tcl、superlint.tcl、DC.sdc。
  • 合成後檔案：drawBbox_syn.v、drawBbox_syn.sdf。
  • 結果圖片與文字檔：.bmp、.txt，依助教規定放入。
  • 正式報告：StudentID.pdf。

2、不要放

  • .fsdb、.vcd 等 waveform 檔案。
  • 不必要的大型暫存檔、工具產生的 cache、無關圖片。
  • test_fig 檔案。附件紅字已特別標註：No need to include test_fig file。
  • 報告中不要整份貼上 source code。

五、提交前檢查清單

建議做法：不要只檢查原本工作資料夾。請把最後的 .tar 解壓縮到一個新的空資料夾，重新執行 compile / simulation / synthesis / SuperLint，確認助教端也能重現結果。
  1. 確認壓縮檔名稱為 Lab7_StudentID.tar，StudentID 已換成自己的學號。
  2. 確認壓縮檔內部資料夾階層正確，不是直接散落一堆檔案。
  3. 確認所有 submodule 都已放入，沒有只放 drawBbox.v。
  4. 確認 vlog 或 Makefile 可以成功編譯所有 Verilog/SystemVerilog 檔案。
  5. 確認 RTL simulation P1、P2、P3、P4 都可以正常跑完。
  6. 確認 synthesis 可以成功產生 drawBbox_syn.v 與 drawBbox_syn.sdf。
  7. 確認 post-synthesis simulation 可以正常執行。
  8. 確認 SuperLint coverage 至少 90%。
  9. 確認報告是 PDF 格式，而且沒有把整份 Verilog source code 貼進去。
  10. 確認沒有放入 .fsdb、.vcd、test_fig 或其他不必要檔案。

六、建議的最終提交指令

在 Linux / SoC Lab 環境中，若資料夾名稱為 Lab7_E24134778，可以使用下列指令建立 tar 檔：

tar -cvf Lab7_E24134778.tar Lab7_E24134778

建立後可用下列指令檢查內容：

tar -tvf Lab7_E24134778.tar

如果檢查時看到資料夾、主 RTL、testbench、合成檔、Makefile、報告 PDF 與所有 submodule 都在裡面，才代表提交包比較安全。

七、附件重點總結

最重要的三件事：
1. 檔案階層要完全正確。
2. 使用到的所有 submodule 都要放入 tar。
3. 不能提交無法編譯的 dead body，也不能放 waveform 檔案。

簡單來說，這份 Deliverables 附件是在提醒學生：Lab7 不只是看報告，也會實際編譯、模擬、合成與檢查 lint。只要缺少檔案或階層錯誤，就可能被扣分；如果整個無法編譯，甚至可能沒有分數。因此最後提交前，務必用全新資料夾重新解壓縮與測試。
